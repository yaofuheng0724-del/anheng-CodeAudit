# AI规则生成增强 - 正反样例输入

## 概述

在现有"AI生成规则"功能基础上增加可选的正反样例描述输入，帮助 LLM 生成更精确的代码检测规则。

## 背景

现有功能：
- 用户输入规则简述（如"检测SQL注入"）
- LLM 生成 Markdown 格式的规则描述文本（包含检测目标、检测原理、检测模式、误报排除、修复建议等）
- 生成的规则用于 LLM 代码审计

增强需求：
- 增加正样例描述输入（什么情况应该报告）- 可选
- 增加反样例描述输入（什么情况不应报告）- 可选
- LLM 根据简述 + 正反样例生成更精确的检测模式

## 设计细节

### 用户输入

| 输入项 | 必填 | 说明 |
|--------|------|------|
| 规则简述 | 是 | 用自然语言简述检测逻辑，如"检测SQL注入" |
| 正样例描述 | 否 | 应该被报告的情况描述，如"使用f-string拼接SQL语句" |
| 反样例描述 | 否 | 不应被报告的情况描述，如"使用ORM参数化查询" |

### UI 设计

采用紧凑布局，在现有"AI生成规则"Tab页中，规则简述下方依次添加两个小输入框（各2-3行）。

布局顺序：
1. 规则简述（必填，现有）
2. 正样例描述（可选，新增）
3. 反样例描述（可选，新增）
4. 生成规则按钮
5. 生成结果文本框

### API 改动

**请求参数增加：**
```typescript
interface RuleGenerateRequest {
  description: string;        // 规则简述（必填）
  positiveExample?: string;   // 正样例描述（可选）
  negativeExample?: string;   // 反样例描述（可选）
  language?: 'zh' | 'en';     // 输出语言
}
```

### 提示词改动

当用户提供正反样例时，系统提示词会额外包含：

```
## 用户提供的样例参考

### 正样例（应该报告的情况）
{positiveExample}

### 反样例（不应报告的情况）
{negativeExample}

请根据上述正反样例，在"检测模式"部分重点覆盖正样例描述的场景，
在"误报排除"部分重点排除反样例描述的场景。
```

## 涉及改动

| 文件 | 改动内容 |
|------|----------|
| `frontend/src/pages/PromptManager.tsx` | 新增状态变量、输入框、传递参数到API |
| `frontend/src/shared/api/aiRules.ts` | 请求参数接口增加样例字段 |
| `backend/app/api/v1/endpoints/ai_rules.py` | 请求模型增加样例字段，传递到服务 |
| `backend/app/services/rule_generator.py` | 函数签名增加样例参数，提示词模板增加样例处理 |

## 实现步骤

1. 后端：修改 `rule_generator.py` 服务
   - 函数签名增加 `positive_example`、`negative_example` 参数
   - 修改系统提示词模板，当样例存在时动态注入

2. 后端：修改 `ai_rules.py` API
   - 请求模型增加样例字段
   - 调用服务时传递样例参数

3. 前端：修改 `aiRules.ts` API
   - 接口增加样例字段
   - 调用时传递样例参数

4. 前端：修改 `PromptManager.tsx`
   - 新增状态变量 `positiveExample`、`negativeExample`
   - 新增两个输入框
   - 调用 API 时传递样例参数

## 验证方式

- 输入规则简述"检测硬编码密码"
- 输入正样例"在代码中直接写密码字符串，如 password='123456'"
- 输入反样例"从环境变量或配置文件读取密码"
- 验证生成的规则在"检测模式"中包含硬编码密码场景，在"误报排除"中排除环境变量读取场景
